# 復現完成條件

本文定義「Table 1-14 論文表格復現完成」需要具備的可檢查證據。

## 不算完成的狀態

以下狀態都不應宣稱已完成復現：

- 只有程式碼與 configs，尚未執行完整 pipeline。
- 只執行 `fixture_smoke`。它只驗證報表鏈，不處理 EEG、不訓練模型。
- 只有舊 runs，且 `test_predictions.csv` 缺少 `prob_*` 欄位。
- 只有 `paper-side` 表格，沒有 ours-side run outputs。
- `audit.md` 中仍有與正式 scenario 相關的 `missing` 或 `stale` 項目。

## 正式完成標準

本專案保留兩個可執行 scenario。若目標是「完全照論文文字方法」復現，正式完成判斷應先看 `paper_literal_80_10_10`；`val_as_test_80_20` 是用來解釋論文 reported support 疑點的 comparison/audit scenario。

| Scenario | 用途 |
|---|---|
| `paper_literal_80_10_10` | 照論文文字描述的 `80/10/10` split 復現流程 |
| `val_as_test_80_20` | 照論文 support 反推的 `80/20 validation-as-test` 流程 |

照論文文字方法完整執行：

```bash
uv run python scripts/ds004504_rbp_paper/run_reproduction_pipeline.py \
  --scenario paper_literal_80_10_10 \
  --execute
```

若尚未下載資料，需加上：

```bash
--include-download
```

執行後必須確認：

```bash
uv run python scripts/ds004504_rbp_paper/audit_reproduction_artifacts.py \
  --scenario paper_literal_80_10_10 \
  --fail-on-missing
```

完整 pipeline 的最後一步已內建同一個 `--fail-on-missing` audit gate；也可以重新執行上面的 command 獨立確認。此 command 退出碼為 `0`，才代表正式 artifacts 齊全。
Audit 會檢查檔案存在與內容覆蓋：`table_summary.csv` 必須涵蓋 Table 1-14，`paper_vs_ours.csv` 必須涵蓋可比較的 Table 3-13，`protocol_manifest.csv` 必須涵蓋資料、切分、模型、optimizer、training runtime 等 protocol components，`issue_summary.csv` 必須涵蓋既定論文疑點。
正式 scenario 中 `missing`、`stale` 與 `not_applicable` 都不應存在；`not_applicable` 只允許在 `fixture_smoke` 中出現。

若要檢查 paper-inferred `80/20 validation-as-test` protocol，使用：

```bash
uv run python scripts/ds004504_rbp_paper/run_reproduction_pipeline.py \
  --scenario val_as_test_80_20 \
  --execute
```

接著：

```bash
uv run python scripts/ds004504_rbp_paper/audit_reproduction_artifacts.py \
  --scenario val_as_test_80_20 \
  --fail-on-missing
```

同樣必須退出碼為 `0`，才代表該 scenario artifacts 齊全。

若 `run_reproduction_pipeline.py` 在 command plan 中標出 `partial-output`，代表某個 training output directory 已存在但 marker 檔不存在。這通常是前一次 run 中斷造成；正式復現前應移除該 output directory 或補完該 run。
`--allow-partial-outputs` 只能用於人工診斷或恢復流程，不應作為正式完成證據。Final verifier 會拒絕 `allow_partial_outputs: true` 的 full wrapper manifest。

若要完成「照論文文字流程」與「support 反推流程」的總比較，兩個 scenario 都完成後執行：

```bash
uv run python scripts/ds004504_rbp_paper/compare_report_scenarios.py \
  --fail-on-missing
```

此 command 退出碼為 `0` 時，代表兩個 scenario 的 `table_summary.csv`、`protocol_manifest.csv`、`paper_vs_ours.csv` 與 `issue_summary.csv` 都存在，且 scenario-level table/protocol/metric/issue coverage 沒有 missing/stale 項目。

也可以用單一 wrapper 執行完整比較流程：

```bash
uv run python scripts/ds004504_rbp_paper/run_full_comparison.py \
  --execute
```

此 wrapper 會寫出：

```text
data/reports/ds004504_rbp_paper/full_comparison_execution_manifest.json
```

Wrapper manifest 也會記錄 final verifier 使用的 coverage CSV path，預設為：

```text
data/reports/ds004504_rbp_paper/scenario_comparison/scenario_coverage.csv
```

Final verifier 會要求 manifest 中記錄的 `coverage` 欄位等於本次 verifier 實際檢查的 coverage CSV path，避免用不同 wrapper run 的 manifest 與 coverage 檔案混合當作完成證據。

若使用 full wrapper，manifest 中的 `workflow_status` 必須是 `completed`，且非 verifier 的 workflow steps 都必須是 `status: completed` 且 `returncode: 0`，才代表 wrapper 層級的復現與比較步驟成功。最終 wrapper 結束後，`overall_status` 也應是 `completed`；verifier 執行中短暫寫入的 `overall_status: verifying` 只代表 final verifier 正在檢查。若 manifest 中已記錄 verifier step，該 step 也必須是 `completed` 且 `returncode: 0`。dry-run 只會記錄 `workflow_status: planned`、`overall_status: planned` 與 `status: planned`，不能作為完成證據。
Final verifier 會要求 manifest 的 `execute` 為 `true` 且 `mode` 為 `EXECUTE`；dry-run 或手動改 status 的 plan manifest 不能作為完成證據。
手動執行 `verify_full_reproduction.py` 時，預設只接受 `overall_status: completed`。full wrapper 內部呼叫 verifier 時會加上 `--allow-verifying`，因為 wrapper 會在啟動 verifier 前先寫出「workflow 已完成、final verifier 正在檢查」的中間 manifest。保存為最終完成證據的 manifest 應該是 `overall_status: completed`。
若 `workflow_status: failed`，代表 paper-literal pipeline、80/20 support-audit pipeline 或 scenario comparison gate 在 verifier 之前失敗。若 `workflow_status: completed` 但 `overall_status: failed`，代表前置 workflow 已完成，但 final verifier 沒通過。
Manifest 中的 `verifier_allow_verifying` 記錄 wrapper 是否用 `--allow-verifying` 呼叫 final verifier。full comparison mode 會是 `true`，因為 wrapper 內部 verifier 需要讀取 `overall_status: verifying` 中間態；`--skip-val-as-test` paper-only mode 不會執行 full verifier，通常是 `false`。
Manifest 中的 `verifier_script` 記錄 final verifier 入口，預期為 `scripts/ds004504_rbp_paper/verify_full_reproduction.py`。
Manifest 中的 `comparison_script` 記錄 scenario comparison 入口，預期為 `scripts/ds004504_rbp_paper/compare_report_scenarios.py`。
Manifest 中的 `pipeline_script` 記錄單一 scenario pipeline 入口，預期為 `scripts/ds004504_rbp_paper/run_reproduction_pipeline.py`。
Manifest 中的 `scenarios` 記錄 wrapper 計畫執行的 scenario。完整比較應包含 `paper_literal_80_10_10` 與 `val_as_test_80_20`；paper-only mode 只會包含 `paper_literal_80_10_10`。
`verify_full_reproduction.py` 會要求 `skip_val_as_test` 為 false，manifest 的 `scenarios` 同時包含 `paper_literal_80_10_10` 與 `val_as_test_80_20`，且 `scenario_coverage.csv` 也必須同時包含這兩個 scenario。
對每個 scenario，`scenario_coverage.csv` 必須包含四類 coverage：`table_summary.csv:paper_table`、`paper_vs_ours.csv:paper_table`、`protocol_manifest.csv:component`、`issue_summary.csv:issue_id`。
完整「復現後再比較兩者」的完成證據必須同時包含：`full_comparison_execution_manifest.json` 的 `workflow_status: completed`、`overall_status: completed` 且 workflow steps 都是 `completed`，以及 `scenario_comparison/scenario_coverage.csv` 沒有 `missing` 或 `stale`。
Final verifier 會要求 `scenario_coverage.csv` 中每個 required scenario 的每個 required coverage artifact 狀態都是 `ok`；任何非 `ok` 狀態都不能作為完成證據。
Final verifier 也會拒絕 `scenario_coverage.csv` 中出現 required scenarios 之外的 scenario，確保 full-comparison 完成證據只代表正式 `paper_literal_80_10_10` 與 `val_as_test_80_20` 兩者比較。
Final verifier 會拒絕同一個 scenario 下重複的 coverage artifact row；每個 scenario/artifact 應只有一個明確的 coverage 結論。
Final verifier 也會檢查 coverage CSV 所在 comparison directory 中的 scenario comparison summary/detail artifacts 都存在且非空，包含 table summary、protocol summary、paper-vs-ours details、issue summary 與 markdown comparison report。Scenario comparison CSV 不只要存在，還必須至少有一筆資料列，且 `scenario` 欄位必須剛好涵蓋 full-comparison required scenarios。

可用以下 command 檢查上述完成證據：

```bash
uv run python scripts/ds004504_rbp_paper/verify_full_reproduction.py
```

`run_full_comparison.py --execute` 在 full mode 會自動執行這個 verifier；手動執行通常用於重新檢查既有 artifacts。

若加上 `--skip-val-as-test`：

```bash
uv run python scripts/ds004504_rbp_paper/run_full_comparison.py \
  --skip-val-as-test \
  --execute
```

這只代表 `paper_literal_80_10_10` 正式復現流程完成，不代表兩個 scenario 的總比較完成。

## 判讀原則

- `paper_vs_ours.csv` 是主要比較表。
- `table_summary.csv` 是每張 Table 的快速摘要，適合先看整體差異。
- `issue_summary.csv` 是論文內部矛盾與復現風險摘要，適合先判斷差異來源。
- `support_comparison.csv` 用來檢查 Table 3-7 的 support 是否接近論文表格與 Figure 4。
- `classification_metrics.csv` 用來檢查每一類 precision / recall / F1 / sensitivity / specificity / support。
- `kfold_accuracy.csv` 用來檢查 Table 10-11。
- `model_parameters.csv` 用來檢查 Table 2。
- `audit.md` 是完成狀態證據；正式 scenario 中不能有 `missing` 或 `stale`。

## Coverage gate 維護規則

Table、paper-vs-ours、protocol 與 issue 覆蓋規則集中定義在：

```text
scripts/ds004504_rbp_paper/report_coverage.py
```

`audit_reproduction_artifacts.py` 與 `compare_report_scenarios.py` 都應引用這個檔案，不要在各自腳本中維護重複常數。
`verify_full_reproduction.py` 也應引用同一個檔案中的 full-comparison scenario 與 coverage artifact 常數，避免 final verifier 與 scenario comparison 對完成條件的定義不同步。
Wrapper manifest 中的 `pipeline_script`、`comparison_script` 與 `verifier_script` 也應由同一個檔案定義並由 final verifier 檢查，避免非預期入口產生的 manifest 被誤判為正式完成證據。
正式復現 scenario 與 support-audit scenario 應以具名常數定義，再組成 full-comparison scenario tuple；不要在 wrapper 中靠 tuple 位置推論 scenario 角色。
Final verifier 應同時檢查 wrapper manifest 的 `scenarios` 欄位與 `commands` 中實際的 `run_reproduction_pipeline.py --scenario ...` 參數，兩者都必須涵蓋完整比較所需 scenarios。
Final verifier 也會要求 wrapper manifest 中記錄的 commands 都以 `uv run python` 啟動，符合本專案 Python 執行規則。
Final verifier 會要求 wrapper manifest 保留完整 `commands` list；缺少 command plan 的 manifest 不能作為完成證據。
Final verifier 會要求 `commands` list 中每筆 entry 都是 object，且 `command` 欄位是完整 argument list；形狀不完整的 manifest 不能作為完成證據。
Final verifier 也會要求每筆 command entry 保留非空的 `command_text`，讓完成 manifest 可直接回溯實際執行命令。
