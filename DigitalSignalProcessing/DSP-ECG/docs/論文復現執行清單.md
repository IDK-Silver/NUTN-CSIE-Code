# 論文復現執行清單

本文列出產生 Khan et al. 2025 Table 1-14 復現 artifacts 的正式執行順序。

## 目標定義

正式復現主線是 `paper_literal_80_10_10`，也就是照論文文字描述的 `80% train / 10% validation / 10% test` 流程。

`val_as_test_80_20` 不是正式照論文文字的方法；它只用來檢查論文 reported support 是否比較接近 `80% train / 20% validation-as-test`。

## Step 1: 列出 paper-literal command plan

```bash
uv run python scripts/ds004504_rbp_paper/run_reproduction_pipeline.py \
  --scenario paper_literal_80_10_10
```

若 plan 中出現 `partial-output`，代表某個 training output directory 已存在但 marker 檔不存在。正式執行前應移除該 incomplete output directory，或補完該 run。

## Step 2: 執行 paper-literal 復現

```bash
uv run python scripts/ds004504_rbp_paper/run_reproduction_pipeline.py \
  --scenario paper_literal_80_10_10 \
  --execute
```

若本機尚未有 OpenNeuro raw dataset，加入 `--include-download`：

```bash
uv run python scripts/ds004504_rbp_paper/run_reproduction_pipeline.py \
  --scenario paper_literal_80_10_10 \
  --include-download \
  --execute
```

完整 pipeline 最後會執行：

```bash
uv run python scripts/ds004504_rbp_paper/audit_reproduction_artifacts.py \
  --scenario paper_literal_80_10_10 \
  --fail-on-missing
```

因此 pipeline 成功結束才代表正式 scenario 的 Table 1-14 artifacts 齊全。

## Step 3: 檢查 paper-literal 輸出

主要報表：

```text
data/reports/ds004504_rbp_paper/paper_literal_80_10_10/tables/
data/reports/ds004504_rbp_paper/paper_literal_80_10_10/paper_tables/report.md
data/reports/ds004504_rbp_paper/paper_literal_80_10_10/audit/audit.md
```

完成判定：

```text
data/reports/ds004504_rbp_paper/paper_literal_80_10_10/audit/audit.json
```

`audit.json` 不應包含 formal scenario 的 `missing`、`stale` 或 `not_applicable` 項目。

## Step 4: 選擇性執行 80/20 疑點比較流程

若要檢查論文 reported support 是否更接近 `80/20 validation-as-test`，再執行：

```bash
uv run python scripts/ds004504_rbp_paper/run_reproduction_pipeline.py \
  --scenario val_as_test_80_20 \
  --execute
```

這不是正式照論文文字流程，而是用於比較與稽核論文表格疑點。

## Step 5: 產生 scenario-level comparison

兩個 scenario 都完成後執行：

```bash
uv run python scripts/ds004504_rbp_paper/compare_report_scenarios.py \
  --fail-on-missing
```

輸出：

```text
data/reports/ds004504_rbp_paper/scenario_comparison/scenario_coverage.csv
data/reports/ds004504_rbp_paper/scenario_comparison/scenario_table_summary.csv
data/reports/ds004504_rbp_paper/scenario_comparison/scenario_protocol_summary.csv
data/reports/ds004504_rbp_paper/scenario_comparison/scenario_paper_vs_ours.csv
data/reports/ds004504_rbp_paper/scenario_comparison/scenario_issue_summary.csv
data/reports/ds004504_rbp_paper/scenario_comparison/scenario_comparison.md
```

`scenario_coverage.csv` 沒有 `missing` 或 `stale` 時，才代表兩個 scenario 的 comparison artifacts 覆蓋完整。

也可以使用單一入口列出完整命令：

```bash
uv run python scripts/ds004504_rbp_paper/run_full_comparison.py
```

這只會列出 command plan，輸出會標示 `Mode: PLAN ONLY`，每行 command 會以前綴 `would run:` 顯示，不會啟動訓練。

確認後執行：

```bash
uv run python scripts/ds004504_rbp_paper/run_full_comparison.py \
  --execute
```

這個入口會依序執行 `paper_literal_80_10_10`、`val_as_test_80_20`、`compare_report_scenarios.py --fail-on-missing`，最後自動執行 `verify_full_reproduction.py`。
Wrapper 內部也會用 `uv run python` 呼叫子命令，不直接使用系統 Python。
執行模式下每個 child command 成功後會印出 `completed`；若流程中斷，最後一個沒有 `completed` 的 step 就是失敗位置。
Wrapper 也會寫出完整比較層級的 command/execution manifest：

```text
data/reports/ds004504_rbp_paper/full_comparison_execution_manifest.json
```

dry-run 會記錄 planned commands；execute 會記錄 completed/failed child steps。
Manifest 會記錄 final verifier 使用的 coverage CSV path，預設是 `data/reports/ds004504_rbp_paper/scenario_comparison/scenario_coverage.csv`。
若要把 wrapper manifest 當作完成證據，`workflow_status` 與 `overall_status` 必須都是 `completed`，且 `execution_results` 裡非 verifier 的 workflow steps 都必須是 `status: completed`。若 manifest 中已記錄 verifier step，該 step 也必須是 `completed`。`overall_status: verifying` 只代表 final verifier 正在檢查；`overall_status: planned` 或 `status: planned` 只代表 dry-run，不代表已完成。
`overall_status: verifying` 是 wrapper 啟動 final verifier 前寫出的正常中間態；wrapper 內部會用 `--allow-verifying` 呼叫 verifier。手動執行 verifier 時預設只接受 `overall_status: completed`。若流程完整結束，wrapper 會再寫回 `overall_status: completed`。
`workflow_status: failed` 表示 verifier 前的 workflow 失敗；`workflow_status: completed` 但 `overall_status: failed` 表示 workflow 完成但 final verifier 沒通過。
完整 comparison 的成功判定還需要搭配 `scenario_comparison/scenario_coverage.csv`；該檔不應有 `missing` 或 `stale`。
也可以用 verifier 做最後檢查：

```bash
uv run python scripts/ds004504_rbp_paper/verify_full_reproduction.py
```

若使用 `run_full_comparison.py --execute` 且沒有加 `--skip-val-as-test`，wrapper 會自動執行這個 verifier；手動 verifier 主要用於重新檢查既有 artifacts。
Verifier 成功輸出的 `recorded_verifier_steps` 代表 manifest 中已記錄的 verifier step 數量；wrapper 內部執行 verifier 時，這個值可能是 `0`，不代表失敗。

若只要執行正式 paper-literal 復現，不做 80/20 疑點流程與 scenario comparison：

```bash
uv run python scripts/ds004504_rbp_paper/run_full_comparison.py \
  --skip-val-as-test \
  --execute
```

此模式只完成正式照論文文字流程，不完成「兩個 scenario 的總比較」。

建議先看：

`report.md` 開頭會標示 scenario role；`paper_literal_80_10_10` 應顯示為正式照論文文字的 `80/10/10` 復現流程。

data/reports/ds004504_rbp_paper/paper_literal_80_10_10/tables/paper_vs_ours.csv
