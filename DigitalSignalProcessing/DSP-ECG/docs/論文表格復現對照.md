# 論文表格復現對照

本文整理 Khan et al. 2025 論文 Table 1-14 在本專案中的復現來源、必要 artifact 與比較輸出。

## 完整 pipeline

本專案把復現分成兩個正式 scenario：

| Scenario | 目的 | 是否代表「照論文文字方法」 |
|---|---|---|
| `paper_literal_80_10_10` | 依照論文文字描述的 `80% train / 10% validation / 10% test` 產生 Table 1-14 所需資訊 | 是 |
| `val_as_test_80_20` | 依照論文 reported support 反推的 `80% train / 20% validation-as-test` 檢查表格矛盾 | 否，是 audit/comparison scenario |

列出完整 command plan：

```bash
uv run python scripts/ds004504_rbp_paper/run_reproduction_pipeline.py \
  --scenario paper_literal_80_10_10
```

執行完整 pipeline：

```bash
uv run python scripts/ds004504_rbp_paper/run_reproduction_pipeline.py \
  --scenario paper_literal_80_10_10 \
  --execute
```

完整 pipeline 會產生：

```text
data/reports/ds004504_rbp_paper/paper_literal_80_10_10/tables/
data/reports/ds004504_rbp_paper/paper_literal_80_10_10/paper_tables/
data/reports/ds004504_rbp_paper/paper_literal_80_10_10/plots/
data/reports/ds004504_rbp_paper/paper_literal_80_10_10/audit/
```

若要檢查「論文實際 reported support 比較像 80/20」的問題，再執行：

```bash
uv run python scripts/ds004504_rbp_paper/run_reproduction_pipeline.py \
  --scenario val_as_test_80_20 \
  --execute
```

若只想測試 Table 1-14 報表鏈是否可跑通，可使用 fixture smoke scenario：

```bash
uv run python scripts/ds004504_rbp_paper/run_reproduction_pipeline.py \
  --scenario fixture_smoke \
  --execute
```

`fixture_smoke` 不處理 EEG、不訓練模型，也不能用來報告論文復現數值；它只用來檢查 CSV、Markdown、SVG、audit 產生流程。
在 `fixture_smoke` 的 audit 中，真實 H5/manifest 會標成 `not_applicable`，不是完整復現缺失。

若同時產生 `paper_literal_80_10_10` 與 `val_as_test_80_20`，可產生 scenario-level comparison：

```bash
uv run python scripts/ds004504_rbp_paper/compare_report_scenarios.py
```

此 command 會輸出：

```text
data/reports/ds004504_rbp_paper/scenario_comparison/scenario_table_summary.csv
data/reports/ds004504_rbp_paper/scenario_comparison/scenario_coverage.csv
data/reports/ds004504_rbp_paper/scenario_comparison/scenario_protocol_summary.csv
data/reports/ds004504_rbp_paper/scenario_comparison/scenario_paper_vs_ours.csv
data/reports/ds004504_rbp_paper/scenario_comparison/scenario_issue_summary.csv
data/reports/ds004504_rbp_paper/scenario_comparison/scenario_comparison.md
```

`scenario_coverage.csv` 用來確認每套 scenario 的 Table、protocol、metric 與 issue 覆蓋是否完整。
`scenario_paper_vs_ours.csv` 用來比較兩套 scenario 各自的 paper value、ours value 與 difference。
`scenario_issue_summary.csv` 用來保留論文內部矛盾與本專案處理方式，避免只看數值而忽略流程差異。

## 共用比較輸出

| 輸出 | 用途 |
|---|---|
| `paper_vs_ours.csv` | 逐 metric 比較 paper value 與 ours value |
| `table_summary.csv` | 每張 Table 的 paper/ours accuracy 與 support 快速摘要 |
| `support_comparison.csv` | 比較 paper table support、paper figure row-sum support 與 ours support |
| `confusion_matrices.csv` | 整理 paper Figure 4 與 ours confusion matrices |
| `history.csv` | 整理 ours training/validation history |
| `run_summary.csv` | 彙整每個 run 的 config、split、accuracy、support |
| `protocol_manifest.csv` | 記錄 scenario 的資料版本、RBP bands、split、normalization、SMOTE、k-fold 與 label-swap audit protocol |
| `issue_summary.csv` | 記錄論文內部矛盾、復現風險與本專案處理方式 |
| `audit/audit.md` | 檢查 Table 1-14 所需 artifacts 是否齊全 |
| `full_comparison_execution_manifest.json` | 記錄完整 wrapper 的 planned/completed/failed child steps |
| `scenario_comparison/scenario_coverage.csv` | 檢查兩個 scenario 的 Table、metric、protocol 與 issue 覆蓋 |

`audit/audit.md` 不只檢查路徑存在，也檢查 `table_summary.csv` 的 Table 1-14 覆蓋、`paper_vs_ours.csv` 的 Table 3-13 可比較項目覆蓋、`protocol_manifest.csv` 的 protocol component 覆蓋，以及 `issue_summary.csv` 的既定 issue 覆蓋。
`paper_tables/report.md` 開頭會標示 scenario role，避免把 `val_as_test_80_20` 誤讀成正式論文文字流程。
上述 coverage 規則集中在 `scripts/ds004504_rbp_paper/report_coverage.py`，供單一 scenario audit 與 scenario comparison 共用。
`scripts/ds004504_rbp_paper/verify_full_reproduction.py` 用來檢查完整 wrapper manifest 與 scenario coverage 是否同時達成完成條件。

## Label-swap audit

論文 Table 3 與 Table 6 的 FTD/Healthy support 強烈暗示這兩個任務可能存在 FTD/Healthy label swap。這不是論文文字方法的一部分，因此不列入 paper-literal 結論；它只用來量化表格疑點。

若使用 paper-literal `80/10/10` scenario，label-swap audit configs 是：

```bash
uv run python scripts/ds004504_rbp_paper/train.py \
  cfgs/ds004504_rbp_paper/label_swap/multiclass.yaml

uv run python scripts/ds004504_rbp_paper/train.py \
  cfgs/ds004504_rbp_paper/label_swap/ftd_vs_healthy.yaml
```

若使用 `val_as_test_80_20` scenario，對應 configs 是：

```bash
uv run python scripts/ds004504_rbp_paper/train.py \
  cfgs/ds004504_rbp_paper/label_swap_80_20/multiclass.yaml

uv run python scripts/ds004504_rbp_paper/train.py \
  cfgs/ds004504_rbp_paper/label_swap_80_20/ftd_vs_healthy.yaml
```

這兩個 runs 會以 `ours_label_swap` source 顯示在 Table 3、Table 6、`classification_metrics.csv` 與 `support_comparison.csv`。它們不是 corrected protocol，而是用來檢查論文表格疑點的 audit protocol。

## 重要限制

- `val_as_test_80_20` 是根據論文 support 反推的 paper-inferred protocol，不是論文文字聲稱的 `80/10/10`。
- `paper_literal_80_10_10` scenario 仍可執行，但其 support 不會接近論文 Table 3-7。
- `paper_literal_80_10_10` 也提供 root-level SMOTE configs：`cfgs/ds004504_rbp_paper/smote/*.yaml`。這些 configs 用於照論文文字 split 產生 Table 8-9 形狀的 ours-side 結果。
- Table 8-9 的 SMOTE flow 目前為 simple SMOTE 實作，因論文未提供程式碼與完整 SMOTE 細節。
- Table 10-11 的 k-fold flow 目前採 epoch-level stratified k-fold，因論文未說明 subject-wise fold。
- 完整完成狀態必須以 `audit/audit.md` 為準；有缺失或 stale prediction CSV 時，不應宣稱完整復現已完成。
